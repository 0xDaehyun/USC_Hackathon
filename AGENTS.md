# 프로젝트 작업 규칙

## 기본 원칙

- 작업 전 `README.md`에서 프로젝트 목표와 요구사항을 확인한다.
- 기존 변경 사항을 임의로 삭제하거나 덮어쓰지 않는다.
- 기능 변경 후 관련 테스트 또는 검증을 수행한다.
- 새로운 의존성을 추가할 때는 목적과 이유를 명확히 기록한다.

## 답변 및 판단 규칙

- 답변이나 구현안을 전달하기 전에 관련 파일, 계산, 공식 문서 또는 실행 결과를 이용해 핵심 전제를 한 번 검증한다.
- 최신 정보, 안전, 법률, 의료, 재난 대응과 관련된 주장은 가능한 경우 권위 있는 최신 출처로 확인한다.
- 검증한 내용, 검증하지 못한 내용, 남아 있는 가정을 구분해서 설명한다.
- 실제로 판단이 달라졌다면 `초기 가설 → 재검토 근거 → 최종 판단` 순서로 변경 이유를 간결하게 공유한다.
- 판단이 달라지지 않았다면 이전 생각을 임의로 만들어내지 않고 검증 결과와 최종 결론만 전달한다.
- 내부의 숨은 사고 과정을 그대로 노출하지 않고, 사용자가 판단을 평가하는 데 필요한 근거와 결론 변화만 제공한다.

## 개발 규칙

- P0 앱은 Next.js, React, TypeScript를 사용하고 TypeScript `strict` 모드를 유지한다.
- 패키지 관리는 npm을 사용하며 의존성을 변경하면 `package-lock.json`도 함께 갱신한다.
- 실제 비밀값은 `.env.local` 등 Git에서 제외된 파일에만 저장하고 `.env.example`에는 변수 이름과 안전한 예시만 둔다.
- 외부 데이터 소스를 추가하거나 변경하면 `data/source-manifest.json`에 권한 방식, 검증 일자, 제한사항을 기록한다.
- Live, Historical Reconstruction, What-if 데이터를 타입과 UI에서 명시적으로 구분한다.
- 공식 데이터와 예시·모델 결과를 같은 근거 수준으로 표시하지 않는다.
- 데이터 계약을 변경하면 JSON Schema, TypeScript 타입, 예시 fixture, 검증 스크립트를 함께 갱신한다.
- 변경 전후 최소 `npm run test`와 `npm run lint`를 실행하고, 배포 가능한 변경은 `npm run build`까지 확인한다.
- 커밋은 하나의 검토 가능한 목적에 집중하고, 무관한 사용자 변경 사항을 함께 포함하지 않는다.
